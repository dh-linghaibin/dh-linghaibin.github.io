coos 手册 阅读

  1. 删除任务

    CoExitTask(); //删除自己

    CoDelTask (taskID) //删除其他任务

  2. 任务状态

    TASK_READY 就绪任务 创建好 但是没有运行

    TASK_RUNNING 任务在执行 占用CPU

    TASK_WAITING 等待状态 等待一个事件的发生

    TASK_DORMANT 静止状态 当一个任务被删除时的状态 用户不会被激活

    任务可以在这四个状态之间被更改

    CoSuspendTask() 转换任务的运行状态或准备状态 和 等待状态

    CoAwakeTask() 切换任务从等待状态到就绪状态

  3. 任务控制块

    用于描述一个任务

    >

    stkPtr :指向当前任务堆栈顶部的指针。coocox库斯允许每个任务有自己的堆栈的大小。在每一个任务切换，库斯保存当前CPU通过栈的运行状态，stkptr指出这样的任务可以回到以前的状态时，CPU运行一次。由于Cortex-M3有16个32位通用寄存器描述CPU的状态，一个任务的最小堆的大小是68字节（4字节用于检测堆栈溢出）

    prio: 您分配的任务优先级。多个任务可以在coocox库斯相同的优先级份额

    state: 任务当前的状态

    taskID: 系统分配的任务ID，由于多个任务可以共享相同的优先级，所以优先级不能用作唯一标识符。我们使用任务ID来区分不同的任务coocox库斯。

    mutexID: 任务等待的互斥ID

    eventID ：任务等待的事件ID

    timeSlice: 任务的时间片

    Stack: 一个指针指向栈底。它可以用来检查堆栈溢出

    Pmail: The message pointer sent to the task.

    waitNext: 下一个任务的事件在等待TCB

    waitPrev: 以往的任务事件等待TCB

    taskFuc：任务函数指针，指向活动任务

    taskStk 指向当前任务堆栈顶部的指针，指向活动任务。

    Pnode: 指向事件标志节点的指针

    delayTick: 前延迟事件与任务处于延迟状态时的时间差

    TCBnext: 下一个TCB当任务在就绪列表/延迟/互斥等待名单列表。列出的任务属于由任务状态和TCB的项目确定。如果当前任务处于就绪状态，则处于就绪列表中。如果是在等待状态，由mutexid和delaytick然后判断：如果mutexid不0xffffffff，它是在互斥等待名单；如果delaytick不0xffffffff，它是在延迟列表

    TCBprev: 以前的TCB当任务在就绪列表/延迟/互斥等待名单列表。任务所属的列表由任务状态和相关事件标志决定。

  4. 临界状态 -中断？

    不同于其他的内核，coocox库斯不关中断处理的关键代码段，但锁调度。因此，库斯有更短的时延中断与别人比较。由于启用中断的时间与系统对实时事件的响应有关，所以它是实时内核开发人员提供的最重要的因素之一。与其他方法相比，通过锁定调度器，可以最大程度地提高系统的实时性。因为coocox库斯管理关键部分禁止任务调度，用户的应用程序不能调用API函数将挂起当前正在执行的任务的关键部分

  5. 中断

    在coocox库斯，中断分为两大类，根据是否调用系统API函数内部或不。对于ISR无关的操作系统，coocox库斯不强迫它做任何事情，你可以像没有一个操作系统。然而，对于ISR调用系统API函数里面，coocox库斯要求你调用相关的函数时，进入或退出中断

    >

    所有的系统API，可以称为中断服务程序开始isr_，如isr_postsem（），isr_postmail（），（）和（）isr_postqueuemail isr_setflag。任何其他API在ISR调用会导致系统混乱。当调用中断服务例程中相应的API函数时，系统需要确定任务调度是否锁定。如果解锁，系统可以正常调用它。否则，系统将向服务请求列表发送相关的服务请求，然后等待调度程序的解锁来响应它。

  6. 任务同步

    任务间的同步是指一个任务只能继续执行后得到另一个任务或中断服务程序发送同步信号。有信号，在coocox库斯实现进程间同步互斥和旗帜。

编程的核心

发表于 2017-08-20 | 分类于 c jishu

编程的核心是数据结构，而不是算法

数据是易变的，逻辑是稳定的

函数表驱动法

字符统计

Booch的《面向对象分析与设计》一书中，提到所有的程序设计语言大概有3个源流：结构化编程；面向对象编程；数据驱动编程.我认为数据驱动编程的本质是“参数化抽象”的思想，不同于OO的“规范化抽象”的思想

  1. 数据驱动编程在网络游戏开发过程中很常用，但是少有人专门提到这个词
  2. 数据驱动编程有很多名字：元编程，解释器/虚拟机，LOP/微语言/DSL等。包括声明式编程、标记语言、甚至所见即所得的拖放控件，都算是数据驱动编程的一种吧
  3. 数据驱动编程可以帮助处理复杂性，和结构化编程、OO 均可相容。(正交的角度)
  4. 将变和不变的部分分离,策略和机制分离,由此联想到的还有 数据和代码的分离 微语言和解释器的分离 被生成代码和代码生成器的分离 更近一步 微内核插件式体系结构
  5. 元编程应该说是更加泛化的数据驱动编程 元编程不是新加入一个间接层 而是退居一步 使得当前的层变成一个间接层 元编程分为静态元编程(编译时)和动态元编程(运行时) 静态元编程本质上是一种 代码生成技术或者编译器技术 动态元编程一般通过解释器(或虚拟机) 加以实现
  6. 数据驱动编程当然也不应该说是”反抽象的”,但的确与”OO抽象”的思维方式是迥然不同 泾渭分明的,如TAOUP一书中所述:在Unix的模块化传统和围绕OO语言发展起来的使用模式之间,存在着紧张的对立关系”应该说数据驱动编程的思路与结构化编程和OO是正交的
    编程和人的关系,更类似一种”跳出三界外,不在五行中”的做法.

​ 人类心智的限制，一切的背后都有人的因素作为依据：

​ a 人同时关注的信息数量：7+-2 (所以要分模块)

​ b 人接收一组新信息的平均时间5s (所以要简单，系统总的模块数不要太多)

​ c 人思维的直观性(人的视觉能力和模糊思维能力)，这意味这两点：

​ A “直”——更善于思考自己能直接接触把玩的东西；(所以要“浅平透”、使用具象的设计，要尽量代码中只有顺直的流程)，

​ B “观”——更善于观图而不是推算逻辑；(所以要表驱动法，数据驱动编程，要UML，要可视化编程——当然MDA是太理想化了)

​ d 人不能持续集中注意力(人在一定的代码行数中产生的bug数量的比例是一定的，所以语言有具有表现力，要体现表达的经济性)

​ 所以要机制与策略分离，要数据和代码分离(数据驱动编程)，要微语言，要DSL，要LOP……

​ e 人是有创造欲，有现实利益心的(只要偶可能总是不够遵从规范，或想创造规范谋利——只要成本能承受，在硬件领域就不行)

​ 另外，开一个有意思的玩笑，Unix编程艺术艺术的英文缩写为TAOUP，我觉得可以理解为UP之TAO——向上抛出之道——将复杂的易变的逻辑作为数据或更高层代码抛给上层！

3.2 函数指针

​ “消息处理”一节示例中的函数指针有点插件结构的味道。可对这些插件进行方便替换，新增，删除，从而改变程序的行为。而这种改变，对事件处理函数的查找又是隔离的(隔离变化)。

​ 函数指针非常有用，但使用时需注意其缺陷：无法检查参数(parameter)和返回值(return value)的类型。因为函数已经退化成指针，而指针不携带这些类型信息。缺少类型检查，当参数或返回值不一致时，可能会造成严重的错误。

​ 例如，定义三个函数，分别具有两个参数：

​ int max(int x, int y) { return x>y?x:y; }

​ int min(int x, int y) { return x<y?x:y; }

​ int add(int x, int y) { return x+y; }

​ 而处理函数却定义为：

​ int process(int x, int y, int (f)()) { return (f)(x, y); }

​ 其中，第三个参数是一个没有参数且返回int型变量的函数指针。但后面却用process(a,b,max)的方式进行调用，max带有两个参数。若编译器未检查出错误，而又不小心将return (f)(x,y);写成return (f)(x);，那么后果可能很严重。

​ 因此在C语言中使用函数指针时，一定要小心”类型陷阱”。

代码大全阅读记录

代码大全阅读记录

  • 最小复杂度

避免做出 聪明的设计，难以理解 易于理解简单的设计

  • 易于维护

提高代码质量

  • 松散耦合

通过应用类型的合理抽象，封装 信息影藏 设计出现关联少的类

  • 可扩展性

在增强系统的功能时 无须破坏 底层结构 改动系统的某一部分 不会影响到其他部分

  • 可重用性

设计的系统组成部分 可以在其他系统中使用 使用代价小

  • 高扇入

大量的类使用摸个给定的类 利用 较低层次上的工具类‘’

  • 低扇出

类里少量 或 适中的使用 其他的类 高扇出（超过约7个）说明一个类大量的使用了其他的类 会使系统变得过于复杂 低扇出的原则都是有益的

  • 可移植性

方便移植到其他环境中去

  • 精简性

设计出的系统没有多余的部分

  • 层次性

尽量保持各个系统各个分阶层的层次性，设计出的系统应该可以在任意层次上观察而不需要进入其他层次

  • 标准技术

依赖的外来的 古怪的东西越多 理解越是头疼 要用标准话 常用的方法 让整个系统给人一种熟悉的感觉

系统层次

  • 不同子系统之间相互通讯的规则

所有的子系统都能与其他子系统通讯 完全失去了把他们分开所带来的好处 应该限制所有的子系统之间的通讯来让每个子系统更有存在意义

在没有任何规则时，热力学第二定律就会发生作用，整个系统的熵将会增加，如果不对系统间的通信石佳任何限制，那么它们之间的通信就会肆意发生

只有当‘确需了解’–最好还有合理的理由时，才应该允许子系统之间的通信。如果你还拿不准的话,那么就应该先对子系统之间的通讯加以限制，等日后需要再开放

为了让系统之间连接简单一动且易于维护，就尽量简化子系统之间的交互关系。最简单的交互关是让一个子系统中包含另一个子系统中的类；而最复杂的的交互关系是让一个子系统中的类继承子另一个子系统的类。

A类 - B类 B类 - C类 C类 - A类 —大忌

  • 常见的子系统
  • 子系统会在系统中反复出现
  • 业务规则

在计算机的法律 规则 政策 以及 过程 比如税率 节假日安排

  • 用户界面

把用户界面和其他部分隔离开，以便用户界面的演化不会破坏程序的其余部分 Most situation 用户界面会使用多个附属子系统或类来处理用户界面，命令行接口，菜单操作，窗体管理，帮助系统。。

  • 数据库访问

把对数据库访问的实时细节隐藏起来，让程序不用关系绝大部分可以不必关系的底层处理细节， 提供有价值的的抽象层，从而减少程序的复杂度， 发生错误时，无须修改程序的主要部分

  • 对系统的依赖性

把对操作系统的依赖因素归到一个子系统里，就如同把对硬件的依赖封装起来一样。 比如开发一个window系统，不要局限于window环境，把对window相关的系统调用都隔离起来，放到一个window接口系统中

  • 分解为类
  • 类与对象的比较

核心 对象与类的区分 对象是指运行期间在程序中实际存在的的具体实体，而类是指在程序源码中存在的静态事物。对象是动态的 在运行期间 所能得到的值和属性

  • 分解成子程序

把每个类细分为子程序 3层定义出类接口 4层的设计将细化出类的私用子程序

完成的定义出类内部的子程序，常常会有助于跟好的里i接类的接口，反过来，这又有助于对类的接口进行精伊布的修改

  • 子程序内部的设计

为每个子程序不止详细的功能，子程序的实际工作通常是由负责该子程序的开发人员完成， 包括 伪代码 选择算法 组织子程序内部的代码块
